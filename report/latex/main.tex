\section{System Overview}

Soundscape is a real-time collaborative music room application that solves the problem of synchronizing music playback across multiple listeners. Users can create or join music rooms, queue songs together, and experience synchronized playback with real-time updates using WebSocket communication. The platform enables friends to listen to the same music simultaneously during virtual gatherings or remote hangouts, maintaining a shared queue with voting mechanisms and instant synchronization across all connected clients. This application addresses the gap between individual streaming services and the need for genuine shared listening experiences.

\subsection{Technology Stack}

\begin{itemize}
    \item \textbf{Frontend:} React 18 with Vite, Socket.IO client for real-time communication
    \item \textbf{Backend:} Express.js with Node.js, MongoDB via Mongoose ODM
    \item \textbf{Database:} MongoDB Atlas (cloud-hosted)
    \item \textbf{Real-time Communication:} Socket.IO for WebSocket support
    \item \textbf{Authentication:} Environment variables for secrets (.env)
    \item \textbf{Process Management:} concurrently for unified startup
\end{itemize}

exports.addToQueue = async (req, res) => {
  try {
    const { songId } = req.body;
    if (!songId) 
      return res.status(400).json({ error: 'songId is required' });
    if (!mongoose.Types.ObjectId.isValid(songId)) {
      return res.status(400).json({ error: 'Invalid songId format' });
    }
    const song = await Song.findById(songId);
    if (!song) 
      return res.status(404).json({ error: 'Song not found' });
    const queue = await Queue.findOneAndUpdate(
      { roomId: req.params.roomId },
      { $push: { songs: { songId } } },
      { new: true }
    ).populate('songs.songId');
    res.json(queue);
  } catch (err) {
    res.status(500).json({ error: 'Failed to add to queue' });
  }
};
\end{lstlisting}

exports.updateRoom = async (req, res) => {
  try {
    const { name } = req.body;
    if (!name) 
      return res.status(400).json({ error: 'Room name is required' });
    const room = await Room.findById(req.params.id);
    if (!room) 
      return res.status(404).json({ error: 'Room not found' });
    // Check for duplicate name
    const existing = await Room.findOne({ name });
    if (existing && existing._id.toString() !== req.params.id) {
      return res.status(409).json({ error: 'Room name already taken' });
    }
    room.name = name;
    await room.save();
    res.json(room);
  } catch (err) {
    res.status(500).json({ error: 'Failed to update room' });
  }
};
\end{lstlisting}

\section{Reflection: Development Challenges and Solutions}

\subsection{Challenge: Real-Time Synchronization Across Multiple Clients}

\textbf{Problem:} The initial approach attempted to rely solely on REST polling for playback state updates. This created several issues:
\begin{itemize}
    \item Network latency caused desynchronization between clients
    \item Excessive database queries from frequent polling
    \item No truly instantaneous updates when users added songs or paused
    \item Poor user experience with noticeable delays
\end{itemize}

\textbf{Solution:} Integrated WebSocket via Socket.IO for bidirectional communication. This enabled:
\begin{itemize}
    \item Instant event broadcasting to all connected room clients
    \item Server-side room state management via Socket.IO namespaces
    \item Event-driven architecture replacing continuous polling
    \item Reduced database load and network overhead
\end{itemize}

\textbf{Implementation Details:}
\begin{itemize}
    \item Socket.IO server listening on room-specific events
    \item Client-side socket listeners automatically sync UI state
    \item Queue changes broadcast immediately without page reload
    \item Playback position updated in real-time across all listeners
\end{itemize}

\textbf{Learning Outcome:} Real-time applications require asynchronous, event-driven communication rather than synchronous polling. Understanding the CAP theorem (eventual consistency vs. strong consistency) became crucial for designing responsive distributed systems. This challenge demonstrated why production applications often combine REST APIs for CRUD operations with WebSockets for live updates.

\subsection{Challenge: MongoDB Atlas Connection via DNS SRV Records}

\textbf{Problem:} The application initially used the `mongodb+srv://` URI scheme for MongoDB Atlas connectivity. During deployment testing, connections failed with `ECONNREFUSED` errors despite correct credentials.

\textbf{Root Cause:} DNS SRV record resolution failed on the target network. The `mongodb+srv://` URI relies on DNS SRV records to discover and connect to MongoDB Atlas cluster nodes. When the DNS server did not return SRV records, the driver could not resolve cluster node addresses. This issue was discovered during initial MongoDB setup (Commit 156b5d82, April 8).

\textbf{Solution:} Switched from SRV-based URI to explicit connection string with all node addresses:

\begin{lstlisting}[language=bash, caption=Connection String Before (SRV)]
mongodb+srv://mongo:1404@soundscape.icmlyru.mongodb.net/?appName=soundscape
\end{lstlisting}

\begin{lstlisting}[language=bash, caption=Connection String After (Explicit Nodes)]
mongodb://mongo:1404@ac-zgqzfqs-shard-00-00.icmlyru.mongodb.net:27017,\
ac-zgqzfqs-shard-00-01.icmlyru.mongodb.net:27017,\
ac-zgqzfqs-shard-00-02.icmlyru.mongodb.net:27017/?ssl=true&\
replicaSet=atlas-61fuhw-shard-0&authSource=admin&appName=soundscape
\end{lstlisting}

\textbf{Key Differences:}
\begin{itemize}
    \item Explicit protocol: `mongodb://` instead of `mongodb+srv://`
    \item All three shard nodes listed with port 27017
    \item Replica set name specified: `atlas-61fuhw-shard-0`
    \item SSL/TLS explicitly enabled
    \item Admin authentication database specified
\end{itemize}

\textbf{Implementation:} Updated `.env` file with explicit connection string, ensuring environment-specific DNS resolution issues don't prevent database connectivity. Connection string documented in Commit 156b5d82 with notation: ``connected to mongo (using standard connection string as SRV did not work, took some hours of troubleshoot)''.

\textbf{Learning Outcome:} Cloud services provide multiple connectivity options for resilience. While SRV records offer simplicity and flexibility, explicit connection strings provide stability in networks with DNS resolution constraints. This experience highlighted the importance of having fallback connection strategies and understanding the underlying protocols in distributed systems.

\section{Feature Iteration and Improvements}

\subsection{Initial Implementation}

The queue system initially stored all songs as flat references without metadata. Additions to the queue were simple array pushes with no tracking of when songs were added or user voting.

\begin{lstlisting}[language=javascript, caption=Early Queue Model (April 8 Initial Backend Setup)]
const queueSchema = new mongoose.Schema({
  roomId: mongoose.Schema.Types.ObjectId,
  songs: [mongoose.Schema.Types.ObjectId]
});
\end{lstlisting}

\textbf{Issues:}
\begin{itemize}
    \item No way to track insertion order with meaningful metadata
    \item Cannot implement voting mechanisms for songs
    \item Lost information about who added what and when
    \item Difficult to implement removal by queue position
\end{itemize}

\subsection{Improved Implementation}

The queue schema was refactored to include an embedded document for each queue entry, capturing metadata crucial for domain-specific features.

\begin{lstlisting}[language=javascript, caption=Enhanced Queue Model (April 20 API Improvements - Commit ee3146c6)]
const queueSchema = new mongoose.Schema({
  roomId: { 
    type: mongoose.Schema.Types.ObjectId, 
    ref: 'Room', 
    required: true 
  },
  songs: [{
    songId: { type: mongoose.Schema.Types.ObjectId, ref: 'Song' },
    addedAt: { type: Date, default: Date.now },
    votes: { type: Number, default: 0 }
  }]
});
\end{lstlisting}

\textbf{Improvements:}
\begin{itemize}
    \item \texttt{addedAt} timestamp enables chronological ordering and UI displays
    \item \texttt{votes} field supports democratic song promotion
    \item Structured as embedded documents for atomic operations
    \item Enables sorting and filtering by addition time or vote count
\end{itemize}

\subsection{Improvement: ObjectId Validation on Queue Additions}

\textbf{Initial Issue:} When users submitted malformed MongoDB ObjectIds to the `POST /api/rooms/:roomId/queue` endpoint, Mongoose threw a `CastError` during database operations. The generic catch block then returned HTTP 500 (Internal Server Error) instead of the semantically correct HTTP 400 (Bad Request).

\textbf{Before:}
\begin{lstlisting}[language=javascript]
const { songId } = req.body;
if (!songId) 
  return res.status(400).json({ error: 'songId is required' });
// Missing: ObjectId format validation
const song = await Song.findById(songId);  // Throws CastError if invalid
\end{lstlisting}

\textbf{After:}
\begin{lstlisting}[language=javascript]
const { songId } = req.body;
if (!songId) 
  return res.status(400).json({ error: 'songId is required' });
if (!mongoose.Types.ObjectId.isValid(songId)) {
  return res.status(400).json({ error: 'Invalid songId format' });
}
const song = await Song.findById(songId);
\end{lstlisting}

\textbf{Impact:}
\begin{itemize}
    \item Prevents unhandled exceptions from propagating
    \item Returns correct HTTP 400 status for client errors
    \item Provides clear error message for debugging
    \item Improves API reliability and client experience
\end{itemize}

\textbf{Pattern Applied:} This validation pattern was added to all routes accepting ObjectId input, ensuring consistent error handling across the API. Implemented in Commit ee3146c6 along with PUT endpoint for room updates and edit UI integration.
